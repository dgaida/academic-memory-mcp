    Database Processes¶

    The system uses two separate SQLite databases to ensure a clean separation between general file metadata/search indices and university-specific organizational personnel and module data:

    1. Central Metadata Database (university.db): Manages file metadata, folder structures, summaries, email conversations, student records, and deadlines.
    2. Personnel and Module Assignments Database (th_personal.db): Manages the organizational hierarchy (faculties, institutes), imported university personnel, as well as module responsibilities and examination regulations (MOCOGI).

    This page describes the processes that read from or write to these two databases.

    Reading Processes¶

    Processes on the Central Database (university.db)¶

    • CLI (mcp-uni db list-*): The commands for listing files, folders, students, etc., read directly from the corresponding tables of university.db.
    • Search Index: During a search, metadata is read from university.db to enrich search results from the vector index (Qdrant) with context information (such as file paths).
    • MCP Agent / FastMCP Server: Agents read information from university.db via the server to answer questions or check the status of projects/theses.
    • Person Profiler: Searches email metadata in university.db to find relevant correspondences for a specific person.

    Processes on the Personnel Database (th_personal.db)¶

    • Knowledge Graph Visualization (python -m th_personal_graph.scripts.visualize_knowledge_graph): The script reads all nodes (persons, modules, POs, faculties) and edges from th_personal.db to generate the interactive HTML graphic.

    Writing Processes¶

    Processes on the Central Database (university.db)¶

    1. Crawler (mcp-uni index)¶

    The crawler is the primary writing process for university.db. It performs the following actions:
    File Metadata: Stores path, hash, modification time, and type of each file found.
     Folder Structure: Records the hierarchy of scanned directories.
    Summaries: Stores the summaries generated by the LLM for files and folders.
     Email Conversations: Groups emails into conversations and stores aggregated summaries.

    2. Watcher (mcp-uni watch)¶

    The watcher monitors the file system in real-time. When changes occur, it performs the same write operations as the crawler for the affected files on university.db.

    3. Knowledge Graph Engine¶

    During indexing, the engine extracts entities and relationships from document summaries and stores them as nodes and edges in the graph structure of university.db.
    Name Resolution: Node names are resolved to their canonical form via the alias table before storing to avoid duplicates.
     Edge Priorities: Based on ontology.yaml, the engine implements priority logic. New relationships can replace existing relationships of the same category between two nodes if they have a higher priority.

    4. Student Sync (mcp-uni db sync-students)¶

    Reads a local students.yaml file and writes the information contained therein about students, their status, and thesis topics into university.db.

    5. Email Controller (scripts/process_sorted_emails.py)¶

    Within the daily email workflow, the controller modifies the data structure:
    Relocation: Physically moves emails on the filesystem upon reclassification.
     Just-in-Time Summary: Generates the .emails_summary.md only during the execution of a reply action to incorporate the most up-to-date context.

    6. Ontology Learner (mcp_university/knowledge_graph/ontology_learner.py)¶

    Automates alias learning for university.db:
    Extracts name-email pairs from email headers.
     Analyzes existing knowledge graph nodes via LLM for module name variations (e.g., "AI" vs "Artificial Intelligence").
    * Stores the results in the aliases table of the MetadataStore.

    Processes on the Personnel Database (th_personal.db)¶

    These scripts are part of the specialized th_personal_graph package and operate in isolation on th_personal.db:

    1. TH Personnel Crawler (python -m th_personal_graph.scripts.crawl_th_koeln_persons)¶

    This script crawls the official personnel directory of TH Köln and writes:
    Person Nodes: Name, email address, and academic degree.
     Organizational Units: Faculties and institutes as nodes.
    Relationships:* Links people to their respective faculties/institutes ("is element of").

    2. MOCOGI Extractor (python -m th_personal_graph.scripts.extract_mocogi_data)¶

    Extracts module information from the MOCOGI API and writes:
    Study Programs: Study programs and examination regulations.
     Modules: All modules of an examination regulation.
    Responsibilities:* Links (matched) people with modules ("is module coordinator", "is first examiner", "is second examiner"). Uses word-set comparison for robust name matching against the imported persons from th_personal.db.

    Deleting Data¶

    Automatic Deletion¶

    The crawler detects during a scan in university.db if registered files no longer exist on the file system. In this case, the corresponding entries in the files table are automatically deleted.

    Manual Deletion¶

    Specific entries can be removed from university.db via the CLI command group mcp-uni db delete-*:
    Files/Folders: Deletes metadata and (in the case of folders, recursively) all contained file references.
     Students/Deadlines: Removes specific records from the management tables.
    Edges:* Allows the removal of relationships in the knowledge graph.

    When deleting a node or a file, the linked summaries may remain as orphaned entries until a cleanup is performed (cleanup process).
