# Database Processes

This page describes the processes that read from or write to the central metadata database (SQLite).

## Reading Processes

Several components access the database in a read-only manner to provide information or make decisions based on it:

*   **CLI (mcp-uni db list-*):** The commands for listing files, folders, students, etc., read directly from the corresponding tables.
*   **Search Index:** During a search, metadata is read from the SQLite database to enrich search results from the vector index (Qdrant) with context information (such as file paths).
*   **MCP Agent / FastMCP Server:** Agents read information from the database via the server to answer questions or check the status of projects/theses.
*   **Knowledge Graph Visualization:** The `visualize_knowledge_graph.py` script reads all nodes and edges to generate the interactive HTML graphic.
*   **Person Profiler:** Searches email metadata to find relevant correspondences for a specific person.

## Writing Processes

The following processes modify the state of the database. This typically happens during indexing or through explicit management commands.

### 1. Crawler (`mcp-uni index`)
The crawler is the primary writing process. It performs the following actions:
*   **File Metadata:** Stores path, hash, modification time, and type of each file found.
*   **Folder Structure:** Records the hierarchy of scanned directories.
*   **Summaries:** Stores the summaries generated by the LLM for files and folders.
*   **Email Conversations:** Groups emails into conversations and stores aggregated summaries.

### 2. Watcher (`mcp-uni watch`)
The watcher monitors the file system in real-time. When changes occur, it performs the same write operations as the crawler for the affected files to keep the database synchronized.

### 3. Knowledge Graph Engine
During indexing, the engine extracts entities and relationships from document summaries and stores them as nodes and edges in the graph structure of the database.
* **Name Resolution:** Node names are resolved to their canonical form via the alias table before storing to avoid duplicates.
* **Edge Priorities:** Based on `ontology.yaml`, the engine implements priority logic. New relationships can replace existing relationships of the same category between two nodes if they have a higher priority.

### 4. Person Crawler (`scripts/crawl_th_koeln_persons.py`)
This script crawls the official personnel page of TH Köln and writes:
*   **Person Nodes:** Name and email address.
*   **Organizational Units:** Faculties and institutes as nodes.
*   **Relationships:** Links people to their respective faculties/institutes ("is element of").

### 5. MOCOGI Extractor (`scripts/extract_mocogi_data.py`)
Extracts module information from the MOCOGI API and writes:
*   **Study Programs:** Study programs and examination regulations.
*   **Modules:** All modules of an examination regulation.
*   **Responsibilities:** Links (matched) people with modules ("is module coordinator", "is first examiner", "is second examiner"). Uses word-set comparison for robust name matching.

### 6. Student Sync (`mcp-uni db sync-students`)
Reads a local `students.yaml` file and writes the information contained therein about students, their status, and thesis topics into the database.

### 7. Ontology Learner (`mcp_university/knowledge_graph/ontology_learner.py`)
Automates alias learning:
* Extracts name-email pairs from email headers.
* Analyzes existing knowledge graph nodes via LLM for module name variations (e.g., "AI" vs "Artificial Intelligence").
* Stores the results in the `aliases` table of the `MetadataStore`.

## Deleting Data

### Automatic Deletion
The **crawler** detects during a scan if files registered in the database no longer exist on the file system. In this case, the corresponding entries in the `files` table are automatically deleted.

### Manual Deletion
Specific entries can be removed via the CLI command group `mcp-uni db delete-*`:
*   **Files/Folders:** Deletes metadata and (in the case of folders, recursively) all contained file references.
*   **Students/Deadlines:** Removes specific records from the management tables.
*   **Edges:** Allows the removal of relationships in the knowledge graph.

When deleting a node or a file, the linked summaries may remain as orphaned entries until a cleanup is performed (cleanup process).
